Git Story

小记
2018-12-10

一、Git概述

Linux 内核开源项目有着为数众多的参与者。整个项目组开始启用一个专有的分布式版本控制系统BitKeeper来管理和维护代码。2005年，开发BitKeeper的商业公司同Linux内核开源社区的合作关系结束，他们收回了 Linux内核社区免费使用BitKeeper的权力。 这就迫使Linux开源社区(特别是Linux的缔造者Linus Torvalds)开发出自己的版本系统。

1.版本控制系统

版本控制系统(VCS)是一种记录一个或若干文件内容变化，以便将来查阅特定版本修订情况的系统。

1).本地版本控制系统。


在本地进行版本管理, 常见的如RCS。

2).集中化的版本控制系统


将版本管理集中于一台单一服务器，该服务器上保存了文件所有的修订版本，开发人员可以从服务器上拷贝工作目录，完成工作后提交到服务器更新修订版本。常见的如CVS、Subversion和Perforce。主要问题是单点故障导致工作阻塞，严重的会导致修订历史版本数据全部丢失。

3).分布式版本控制系统


分布式管理控制系统中每个节点都包含整个仓库的拷贝，如Git。

2.文件存储方式

  • SVN等传统VCS的文件存储方式:记录版本间的增量
  • Git的文件存储方式:记录每个版本的快照

    对于Git的文件存储方式，这样的描述其实并不准确，Git是混合了增量和快照的方式。默认采用快照的方式，当满足一些条件的时候(如git push)，将会对快照进行处理，并计算相似快照的增量进行存储。所以总的来说，通常git仓库会比传统VCS的仓库文件大。这样的存储方式对于Git分支及其管理有较大的益处。

3.Git添加数据问题

在Git中进行提交的数据就一定存在与仓库中，就算将其删除文件其实仍然保存与Git仓库中，可以随时进行恢复。如果误操作将一个较大的文件存储到Git中，虽然可以做到将其从Git仓库中删除，但是较为繁琐。

4.三种区域

  • 工作目录(work directory), 用户的磁盘工作目录
  • 暂存区(stage/index), 保存了下次将提交的文件列表信息
  • Git仓库(repo), 用来保存项目的元数据和提交的数据的地方, 提交的数据都是可以恢复的

4.Git配置

Git配置是KV存储结构, 并采用了分级方式，每一个级别覆盖上一个级别的同名配置:

  • 设置配置
    • git config –system <key> <value>
    • git config –global <key> <value>
    • git config <key> <value>
  • 读取配置
    • git config –system <key>
    • git config –global <key>
    • git config <key>
  • 遍历配置项 : --list

二、Git基础

1.初始化仓库

git init, 该命令将会在当前目录中创建.git/目录, 该目录是Git仓库的入口，git仓库上可以进行嵌套创建的。

git clone <uri>, 若已知远程仓库的uri,并且用户有读权限, 则可以直接clone。

2.跟踪并添加到stage

  • git add ., 将当前目录下的所有文件纳入跟踪和stage。
  • git add <file>, 将指定文件纳入跟踪和stage。
  • git commit -m”message”
  • git commit -a -m”message”

3.其他命令

1).git log

该命令可以查询当前分支上的提交历史, 默认不用任何参数的话，git log会按提交时间列出所有的更新，最近的更新排在最上面。带上参数git log有更强的功能:

  • git log --pretty, 简化显示, 以及进行格式化显示
  • git log -p -<num>, 显示num次数个提交，并且显示每次提交进行的修改。
  • git log --graph, 显示 ASCII 图形表示的分支合并历史。

2).git diff

git diff类似于svn diff，用来显示修改前后的区别，但是git有工作目录, storage, repo三个区域的概念，因此git diff被设计来支持3个区域之间的比较:

  • git diff, 比较当前工作目录和暂存区之间的差异。
  • git diff –cache, 比较暂存区和git仓库之间的差异。
  • git diff HEAD, 比较工作目录和git仓库之间的差异。
  • git diff <commit>, 比较工作目录和某次提交之间的差异。
  • git diff <commit1> <commit2>, 比较工作目录和某次提交之间的差异。

3).撤销修改

git reset类似于svn revert， 可以对更改进行撤销，但是git的reset指令能力更强。先看下reset最简单的几个指令:

  • git reset, 撤销对暂存区的修改, 粒度最小为文件。
  • git reset –hard, 撤销对工作目录的修改， 不能带路径。

git reset命令本质有两个操作：

  • 使用指定的提交快照来覆盖存储区或工作目录, 默认为当前的HEAD所指的快照。
  • 将当前HEAD和分支引用移动至目标提交快照, 默认为当前HEAD所指的快照, 也就是不移动。

需要注意, git reset --hard相当危险，它会用最新的提交快照覆盖掉暂存区和工作目录, 如果希望带路径进行撤销修改可以使用命令git checkout -- <path>，该指令可以用暂存区覆盖工作目录。

4).git tag

可以在HEAD所指的提交上制作一个特殊的引用，这个引用就是tag标签，通常在发布版本的时候会打上标签。

  • git tag, 列出当前所有的tag
  • git tag <new-tag-name> [-m "message"], 在当前HEAD处打上标签。
  • git tag -a <new-tag-name> <commit>, 在知道提交对象hash情况下，可以对指定的提交打上标签。

需要注意，在使用远程仓库时，标签不会自动推送到服务器上，需要显示进行推送, 例如git push origin <tag-name>。

三、Git分支

1.提交对象

每次commit都会在Git仓库中生成一个提交对象, 每个提交对象有3个重要的数据,

  • 提交对象的hash值, 唯一标识了Git仓库中的一个提交对象
  • 当前提交文件的快照片
  • 前一次提交对象的hash值

Git的分支，其实本质上仅仅是指向提交对象的指针，Git的默认分支名字是master。Git中可以创建很多分支，本质上就是创建一个指向提交对象的指针，并通过一个名为HEAD的特殊指针，这个指针可以指向分支名，也可以指向提交对象，指示当前所处的分支。每次提交都会让当前分支的指针指向新的提交对象。

2.常用命令

  • git branch <new-branch>
  • git branch -v
  • git branch -d/-D <branch>
  • git checkout <branch>

3.Merge合并

git merge <branch>, 将<branch>合并到当前分支。

四、Git远程仓库

Git可以添加多个远程仓库，并将版本push到远程仓库或是pull到本地。每个远程仓库的分支在本地也会映射到一个分支，并且分支名为<remote-name>/<branch-name>。远程仓库映射到本地的分支是只读的，表现为若在该分支上进行提交，分支是不会指向新的提交对象。只有pull的时候远程仓库在本地映射的分支才会改变。

1.常用命令

  • git remote -v, 列出远程仓库。
  • git remote add <remote-server-name> <uri>, 添加远程仓库。
  • git remote remove <remote-server-name>, 移除远程仓库。
  • git clone <uri>, 从远程仓库clone, 会自动将其添加到远程仓库。

    2.拉取合并

    拉取远程仓库分支，这一行为本身和自己的工作分支是独立的，拉去远程仓库会更新远处仓库在本地的映射，git pull会自动将远程仓库在本地的映射进行合并，git fetch不会自动合并。
  • git pull
  • git fetch

    3.推送合并

  • git push, 会将本地的分支直接推送到远程仓库的分支上，并在远程仓库进行合并，若合并失败会返回客户端失败，并建议客户端拉去服务器分支进行合并后再push。

五、Git常用工作流

1.特性分支

特性分支是一种短期分支，通常是为了完成一个需求或是解决一个Bug而建立。特性分支完成后，会将其合并回master分支。

2.集中式工作流及分支模型

通常在企业采用集中式工作流程，这和SVN中的工作模式比较类似。

该工作流的分支模型多变，通常集中式的远程仓库中可能会有以下分支

  • mater, 正式环境分支
  • test, 测试环境分支
  • dev, 开发分支
  • issue1, 需要多人进行开发的迭代分支
  • arthurlu, 开发人员分支

远程仓库可能有成千上万的分支, 开发完成后要合并到master分支或是test分支需要发起merge request，这是一种code review，经同意后才会进行合并。

3.集成管理者工作流

这也是GitHub开源项目进行贡献的核心流程。

会有一个管理者远程仓库(blessed), 该远程仓库只能由管理中读写, 其他开发人员只有读权限。各个开发人员可以fork这个远程仓库到自己的远程仓库中，再将自己的远程仓库clone到本地进行开发，开发完成后上传到开发人员自己的远程仓库。开发人员用远程仓库发起pull request，请求管理者将其开发的特性/分支合并到管理者的仓库中。

六、Git底层机制

1.Git目录

git init会在当前目录下生成一个.git的隐藏目录，该目录就是本地Git仓库的入口。该目录下包含了一下子目录和文件：

Git除了merge, checkout, rebase等等应用层命令外，还有一系列的底层

2.Git对象

1).Blob对象和Git的KV操作

Git采用kv对数据库, 在db中存入value, db将会分配一个key，通过该键值，可以任意适合进行检索。其实key就是value的类hash, value不变则key不变。

  • echo "<value>" | git hash-object -w –stdin
  • git has-object -w <file>, 将<file>中的数据作为value保存, git add本质上就是做这样的操作。
  • git cat-file -p <key>, 可以获取对应<key>的value。

生成的kv对象将会生成在.git/objects/目录下，每个kv对就是一个该目录下的文件。这样的对象在Git中被称为blob对象

2).tree object

需要注意的是blob中仅仅保存了文件的数据，并没有保存文件的名称等信息。通过tree对象，可以组织文件的名字，也可以将多个文件组织在一起，这是一个类似与目录结构的对象。每个tree object都包含了一个或多个tree entry, 每个tree entry中包含了文件信息以及一个指向blob或子tree的hash指针。

  • git cat-file -p master^{tree}, master^{tree}表示master分支所指向的树对象(每次提交都会指向一个树对象)
  • git update-index –add –cacheinfo <mode> <file-hash> <file>, 将文件放到暂存区
  • git write-tree, 将暂存区内容写入一个树对象

    3).commit object

    每次commit命令，都会将暂存区生成一个树对象，再用一个提交对象指向该树对象。可以通过底层指令实现提交。
  • echo "message" | git commit-tree <tree object hash> [-p <parent>]

    3.打包

    众所周知，Git每次提交都会建立一次快照，会随着时间的累计导致Git仓库非常庞大，因此需要将Git进行压缩打包，Git压缩打包本质上是整理Git的版本数据和垃圾回收，使用命令git gc。git的所有对象数据都放在.git/objects/目录下，当执行了打包，会将数据压缩并放到.git/objects/pack目录下，这个目录下有打包的相关数据信息。打包会将相似数据进行增量编码，并将版本库中不可达的对象删除。

七、相比与SVN的优缺点

优点

  • 分布式版本管理, git可以避免将一些本地分支对外开放, 也具备更好的可用性。
  • 轻量级分支, svn的分支基于服务器目录+写时复制(老版本的svn基于目录拷贝), git的分支是基于全量记录+标识的。
  • 储存功能
  • 关键词定位

缺点

  • 全局唯一版本号, 可以通过hook缓解该问题
  • 读写锁，git不支持写锁申请，不便于二进制文件的管理
  • 全量记录
  • 权限管理粒度
  • clone代价

附录一、SVN和Git的混合开发模式

Git越来越流行，受到各个开发人员的追捧，但一些历史项目仍然采用了subversion, cvs等版本管理。项目迁移成本大，git又具有一定的学习成本，并不需要为了追求Git而把整个项目迁移。在避免迁移的基础上，可以采用混合开发模式，开发人员可以根据自己的情况将git的管理优势引入到开发中。

混合开发模型

特点

  • 分布式Git, 本地项目管理。
  • 轻量级分支, 快速在本地建立分支, 业务并行处理。
  • Git合并, 将svn本地和svn服务器的代码冲突交给Git进行处理。
    • svn的冲突会生成3个文件，手动解决冲突后, 需要进行删除(svn resolved), 才能进行commit。
  • 版本发布的Tag标签。
  • Git储存, 将不确定的修改进行储存。
  • 回退修改。
